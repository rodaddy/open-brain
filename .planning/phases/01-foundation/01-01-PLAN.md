**src/db/pool.ts:** Create the PostgreSQL connection pool factory.
- Export `createPool()` that returns a `pg.Pool` with config from env vars (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD)
- Pool config: `max: 10`, `idleTimeoutMillis: 30000`, `connectionTimeoutMillis: 5000`, `maxUses: 7500`
- Register pgvector types on EVERY new connection: `pool.on("connect", async (client) => { await pgvector.registerTypes(client); })`
- Handle pool errors: `pool.on("error", ...)` logs but does not crash
- Export `checkPoolHealth(pool)` that runs `SELECT 1` and returns `PoolHealth` object with connected/total/idle/waiting
- Import `PoolHealth` from `../types.ts`
- Import pgvector from `"pgvector/pg"`
- Import pg from `"pg"`

**src/db/migrations/001_init.sql:** The complete schema for all 5 tables. Copy EXACTLY from the research (01-RESEARCH.md Code Examples section) -- it has the corrected halfvec_cosine_ops operator class. The SQL creates:
1. `vector` and `uuid-ossp` extensions
2. `thoughts` table with halfvec(768) embedding, content_hash, HNSW index with halfvec_cosine_ops
3. `decisions` table with title, rationale, alternatives (JSONB), context, same embedding pattern
4. `relationships` table with person_name (unique), warmth (1-5 check), last_contact, same embedding pattern
5. `projects` table with name (unique), status, description, tags, metadata (JSONB), same embedding pattern
6. `sessions` table with project, summary, blockers, next_steps, key_decisions (all TEXT[]), same embedding pattern
7. `_migrations` tracking table
8. `ALTER DATABASE open_brain SET statement_timeout = '30s'`

CRITICAL: Use `halfvec_cosine_ops` for ALL HNSW indexes, NOT `vector_cosine_ops`. Use `halfvec(768)` for embedding columns, NOT `vector(768)`.

**src/db/migrate.ts:** A simple migration runner module (~60 lines).
- Export `runMigrations(pool)` that:
  1. Creates `_migrations` table if not exists
  2. Reads all `.sql` files from `src/db/migrations/` sorted by filename
  3. Checks each against `_migrations` table to skip already-applied
  4. Runs unapplied migrations in a transaction (BEGIN/COMMIT per file)
  5. Records each applied migration in `_migrations`
  6. Logs each applied migration via logger
- Use `import { readdir, readFile } from "node:fs/promises"` and `import { join, dirname } from "node:path"`

**scripts/migrate.ts:** CLI entry point that creates a pool, calls `runMigrations`, logs results, then calls `pool.end()`. Use `import.meta.dir` to resolve paths. Exit 0 on success, exit 1 on error.

Create `src/db/migrations/001_init.test.ts` -- an integration test that runs against a real PostgreSQL test database to prove the migration SQL is correct. TypeScript compilation cannot validate SQL correctness; this test catches wrong operator classes, missing tables, incorrect column types, and broken indexes.

**Test setup:**
- Use `beforeAll` to create a test pool pointing at the test database. Read DB connection from env vars but override `DB_NAME` to `open_brain_test` (isolate from production data). If `DB_NAME_TEST` env var is set, use that instead.
- In `beforeAll`, drop all tables (CASCADE) to ensure a clean slate, then run `runMigrations(pool)` to apply the migration fresh.
- In `afterAll`, drop all tables again (clean up) and call `pool.end()`.

**Table existence tests:**
- Query `information_schema.tables` where `table_schema = 'public'`. Assert all 5 tables plus `_migrations` exist.

**Embedding column tests (for each of the 5 tables):**
- Query `pg_attribute` joined with `pg_type` for columns named `embedding`. Assert `typname = 'halfvec'`. Assert the dimension is 768 by checking `atttypmod` (pgvector stores dimension as atttypmod, value = dimension + 4 for halfvec, so expect 772) or by inserting a test row with a 768-dim vector and verifying it succeeds while a 769-dim vector is rejected.

**HNSW index tests:**
- Query `pg_indexes` for indexes containing `hnsw`. Assert 5 indexes exist (one per table). Assert each index definition contains `halfvec_cosine_ops` (NOT `vector_cosine_ops`).

**Content hash unique index tests:**
- Query `pg_indexes` for indexes on `content_hash`. Assert at least 4 exist (thoughts, decisions, relationships, projects -- sessions may or may not have one depending on schema). Verify they are UNIQUE by checking `pg_index.indisunique`.

**Projects table column tests:**
- Query `information_schema.columns` for the projects table. Assert columns exist: `name` (text or character varying), `status` (text), `description` (text), `tags` (ARRAY), `metadata` (jsonb). Assert `name` has a unique constraint (query `pg_constraint` or `pg_indexes`).

**_migrations table test:**
- Query `_migrations` table. Assert it has at least 1 row. Assert the row contains `001_init.sql` in the filename column.

**Vector round-trip test:**
- Insert a row into the `thoughts` table with a known embedding (768-element array of 0.1 values), a dummy content and content_hash.
- SELECT the row back. Assert the embedding is returned as a `number[]` (not a string), and its length is 768. This proves pgvector `registerTypes` works correctly via the pool's `connect` event.
- Delete the test row in cleanup.

**Important:** This test requires a real PostgreSQL instance with pgvector. If the database is unreachable, the test should skip gracefully (use `describe.skipIf` or a try/catch in beforeAll that marks tests as skipped). Add a comment at the top: `// Integration test -- requires PostgreSQL with pgvector. Set DB_NAME_TEST or uses open_brain_test.`
