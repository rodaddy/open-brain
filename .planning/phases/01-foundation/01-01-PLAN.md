**src/db/pool.ts:** Create the PostgreSQL connection pool factory.
- Export `createPool()` that returns a `pg.Pool` with config from env vars (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD)
- Pool config: `max: 10`, `idleTimeoutMillis: 30000`, `connectionTimeoutMillis: 5000`, `maxUses: 7500`
- Register pgvector types on EVERY new connection: `pool.on("connect", async (client) => { await pgvector.registerTypes(client); })`
- Handle pool errors: `pool.on("error", ...)` logs but does not crash
- Export `checkPoolHealth(pool)` that runs `SELECT 1` and returns `PoolHealth` object with connected/total/idle/waiting
- Import `PoolHealth` from `../types.ts`
- Import pgvector from `"pgvector/pg"`
- Import pg from `"pg"`

**src/db/migrations/001_init.sql:** The complete schema for all 5 tables. Copy EXACTLY from the research (01-RESEARCH.md Code Examples section) -- it has the corrected halfvec_cosine_ops operator class. The SQL creates:
1. `vector` and `uuid-ossp` extensions
2. `thoughts` table with halfvec(768) embedding, content_hash, HNSW index with halfvec_cosine_ops
3. `decisions` table with title, rationale, alternatives (JSONB), context, same embedding pattern
4. `relationships` table with person_name (unique), warmth (1-5 check), last_contact, same embedding pattern
5. `projects` table with name (unique), status, description, tags, metadata (JSONB), same embedding pattern
6. `sessions` table with project, summary, blockers, next_steps, key_decisions (all TEXT[]), same embedding pattern
7. `_migrations` tracking table
8. `ALTER DATABASE open_brain SET statement_timeout = '30s'`

CRITICAL: Use `halfvec_cosine_ops` for ALL HNSW indexes, NOT `vector_cosine_ops`. Use `halfvec(768)` for embedding columns, NOT `vector(768)`.

**src/db/migrate.ts:** A simple migration runner module (~60 lines).
- Export `runMigrations(pool)` that:
  1. Creates `_migrations` table if not exists
  2. Reads all `.sql` files from `src/db/migrations/` sorted by filename
  3. Checks each against `_migrations` table to skip already-applied
  4. Runs unapplied migrations in a transaction (BEGIN/COMMIT per file)
  5. Records each applied migration in `_migrations`
  6. Logs each applied migration via logger
- Use `import { readdir, readFile } from "node:fs/promises"` and `import { join, dirname } from "node:path"`

**scripts/migrate.ts:** CLI entry point that creates a pool, calls `runMigrations`, logs results, then calls `pool.end()`. Use `import.meta.dir` to resolve paths. Exit 0 on success, exit 1 on error.
