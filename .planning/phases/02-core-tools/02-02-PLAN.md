  <action>
    1. Create `src/tools/search-brain.ts`:
       - Export `registerSearchBrain(server: McpServer, deps: ToolDeps): void`
       - inputSchema: `{ query: z.string().min(1), table: z.enum([...]).optional(), limit: z.number().int().min(1).max(50).optional() }`
       - Handler logic:
         a. Check auth exists, resolve limit with `args.limit ?? 10`
         b. Determine accessible tables: iterate over all 5 Table values, filter by `canRead(auth.role, table)`. If `args.table` is specified, check it's in the accessible set -- if not, return isError.
         c. If no tables are accessible (discord role), return isError "Permission denied: no readable tables"
         d. Generate query embedding via `deps.embedFn(args.query)`. If null, return isError "Failed to generate query embedding" (unlike writes, search REQUIRES an embedding)
         e. Build the CTE SQL dynamically based on accessible tables. For each accessible table, generate a CTE block following the pattern in 02-RESEARCH.md Pattern 2. Use parameterized queries -- `$1` for the query embedding (toSql'd), `$2` for limit. Do NOT use `$3` table filter parameter -- instead, only include CTEs for tables that pass the permission + filter check. This is safer than the `$3 IS NULL OR $3 = 'tablename'` pattern because it enforces permissions at query construction time.
         f. UNION ALL the CTE results, ORDER BY distance ASC, LIMIT $2
         g. Return results as JSON array in MCP text content format
       - Content preview normalization per table (in CTE SELECT):
         - thoughts: `t.content`
         - decisions: `d.title || ': ' || d.rationale`
         - relationships: `r.person_name || ': ' || COALESCE(r.context, '')`
         - projects: `p.name || ': ' || COALESCE(p.description, '')`
         - sessions: `COALESCE(s.project || ': ', '') || LEFT(s.summary, 200)`
       - Source type labels: "thought", "decision", "relationship", "project", "session" (singular)
       - Each CTE must include `WHERE embedding IS NOT NULL` to avoid NULL distance values
       - Each CTE has its own `ORDER BY distance LIMIT $2` for HNSW index efficiency
       - Use `import { toSql } from "pgvector/pg"` for the query embedding parameter

Test permission filtering by checking which CTEs appear in the SQL string passed to pool.query. Test the error cases (no auth, no read permission, embed failure, empty results).
       - Append to `src/tools/__tests__/protocol.test.ts`: Add search_brain protocol tests in a new describe block. Register search_brain alongside the write tools. Test happy path, empty query validation, and permission denied.

    IMPORTANT: Build the SQL string dynamically (not with $3 filter param) because permission filtering MUST happen at query construction time, not at database level. This prevents data leakage to unauthorized roles. The SQL will vary based on the caller's permissions -- this is intentional and correct.
