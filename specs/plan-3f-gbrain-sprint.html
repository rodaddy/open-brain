Problem

Open Brain already stores entities and typed links, but current search ranking is still mostly vector/keyword first. The link graph is attached as context after retrieval rather than used as a candidate source. That means questions whose answer lives in a relation can miss even when the relation is already in the database.

The risk is over-correction. A broad graph rewrite would add a second substrate, expand the privacy blast radius, and muddy existing namespace guarantees. The sprint has to keep the graph arm optional, bounded, and test-proven.

Solution

Run the sprint as paired issues. Pair A is mandatory: #266 builds the eval gate and #267 implements graph-powered retrieval. Pair B exposes graph-expanded evidence only after Pair A works. Pair C hardens operator visibility and can run in parallel when it does not block Pair A.
